# LUCA Reference Architecture v1
## The Replicable Sovereign AI Agent Framework for Health, Wealth, and Sovereignty

**Version:** 1.0 Reframed  
**Primary live prototype:** Aura Holistic Dental Clinic  
**Ecosystem direction:** Solaris Holistic Health  
**Purpose:** Give people, clinics, businesses, homes, and communities a secure, private, local-first way to organize identity, data, health, wealth, growth, AI agents, workflows, and value exchange.

---

## 1. The clearest way to visualize LUCA

LUCA begins as one local sovereign node powering one real business: Aura Dental.

From there, it becomes a repeatable pattern.

```text
One person
  -> sovereign digital passport
  -> local AI agent
  -> clinic/business/home node
  -> repeatable Clinic-in-a-Box blueprint
  -> many clinics/businesses/homes
  -> regional Solaris nodes
  -> community AI compute
  -> global health, wealth, and sovereignty network
```

The first proof is not theoretical. It is Aura Dental operating as the first live node.

Aura becomes the place where LUCA is tested in real workflows: patient intake, treatment planning, practitioner assistance, local knowledge base, documents, consent, scheduling, follow-up, education, payments, reputation, and operational automation.

This is the beginning of the Solaris network: local first, then regional, then global.

## 5. The LUCA layered architecture

The architecture should be built in layers so that each part can evolve without breaking the whole system.

```text
Layer 10 - Solaris Network and Regional Nodes
Layer 9  - Marketplace, Reputation, and Contribution Network
Layer 8  - Business/Clinic/Home Applications
Layer 7  - AI Agent Runtime and Automation
Layer 6  - Consent, Policy, Compliance, and Audit
Layer 5  - Data Vault, FHIR, Knowledge Graph, and Memory
Layer 4  - Wallet, Bitcoin, Lightning, USDT, and GPS Splits
Layer 3  - Identity, Credentials, KYC/KYA, and ZK Proofs
Layer 2  - Secure Operating System, Containers, and Backups
Layer 1  - Hardware: Personal, Clinic, Regional, and Community Compute
```

Each layer should be replaceable, upgradeable, and modular.

The goal is not to lock LUCA into one vendor. The goal is to create a reference framework where the best available tools can be swapped in and out while preserving the same principles: identity-first, local-first, secure, private, interoperable, and compliance-ready.

## 7. Layer 2 - Secure operating system and local infrastructure

The local node should be simple enough to deploy, but strong enough to trust.

Recommended foundation:

- Linux server base
- Encrypted disk
- Containerized services
- Automatic updates with controlled approvals
- Local firewall
- WireGuard or similar secure remote access
- Password manager
- Hardware security key support
- Local backups
- Offsite encrypted backups
- Disaster recovery plan

Practical rule:

No clinic node should depend on one laptop, one login, one cloud account, or one person remembering where things are stored.

The node needs structure, permissions, backups, logs, and recovery.

## 8. Layer 3 - Identity, credentials, KYC, KYA, and zero-knowledge proofs

Identity is the root of the LUCA system.

The architecture should support multiple identity levels.

### Basic identity

Used for simple users and members.

Includes:

- Name
- Contact information
- Passkey login
- Device registration
- Recovery method
- Consent preferences

### Verified identity

Used when higher trust is needed.

Includes:

- Government ID verification when required
- Phone/email verification
- Proof of address when required
- Anti-fraud checks when required

### Professional identity

Used for practitioners and service providers.

Includes:

- License verification
- Credential verification
- Professional role
- Scope of practice
- Clinic affiliation
- Insurance or compliance documents where applicable

### Agent identity: KYA - Know Your Agent

The agentic web needs KYA.

Every AI agent should have an identity record.

The record should include:

- Agent name
- Owner/controller
- Purpose
- Approved tools
- Data access level
- Spending limits
- Allowed workflows
- Model used
- Model version
- Prompt/policy hash
- Logs of actions
- Human supervisor
- Expiration or review date

This matters because future systems will not only ask, "Who is this user?" They will ask, "Which agent acted, under whose authority, with what permission, and using which model?"

### When KYC is needed

KYC should be used when legal, financial, or regulated workflows require it.

Examples:

- Fiat on/off ramps
- USDT settlement involving regulated providers
- High-value payments
- Practitioner payouts
- Insurance workflows
- Legal agreements
- Medical tourism contracts
- Cross-border payments
- Regulated health services
- Professional licensing verification

### When KYA is needed

KYA should be used when an AI agent can act beyond simple conversation.

Examples:

- Sending messages
- Accessing patient/customer records
- Creating drafts
- Booking appointments
- Recommending operational actions
- Triggering payments
- Updating documents
- Calling APIs
- Managing workflows
- Acting on behalf of a practitioner, clinic, or user

### Where zero-knowledge proofs fit

Zero-knowledge proofs should be used when the system needs to verify something without exposing unnecessary private information.

Examples:

- Prove a person is over a certain age without revealing full ID
- Prove a practitioner has a valid credential without exposing the entire license document
- Prove membership status without exposing full profile
- Prove a user passed KYC without sharing the KYC file with every service
- Prove eligibility for a benefit without exposing all health or financial data

This aligns with tools such as self.xyz-style identity proofs and the broader direction of verifiable credentials.

## 9. Layer 4 - Wallet, Bitcoin, Lightning, USDT, and GPS payment splits

LUCA should treat payments as part of the sovereignty layer.

The wallet layer should support:

- Bitcoin
- Lightning Network
- Stablecoin rails such as USDT where appropriate
- User-configured payment splits
- Contributor payouts
- Clinic payments
- Memberships
- Tips
- Value-for-value flows
- Discounts and incentives
- Audit records for payments

### Why Bitcoin and Lightning matter

Bitcoin and Lightning support fast, open, global value transfer without depending entirely on traditional banks.

This is important for:

- Medical tourism
- International patients
- Cross-border contributors
- Small payments
- Value-for-value models
- Sovereign savings
- Global payment accessibility

### Why USDT can matter

USDT can serve as a bridge to the traditional financial system and to users who think in dollars.

A WDK/Spark-style wallet module can be explored as part of a practical bridge between Bitcoin-native rails and dollar-denominated settlement.

### GPS: Giving / Growth / Prosperity Split

GPS should become a programmable value-routing feature.

Example:

```text
Patient pays $1,000
  -> 80% clinic/provider
  -> 10% Solaris coordination layer
  -> 5% contributor/referral/reputation pool
  -> 5% community/regenerative fund
```

The exact split should be configurable by the user, clinic, or ecosystem rules.

The key idea: value should flow automatically to the people and nodes that create value.

## 11. Layer 6 - Consent, policy, compliance, and audit

This layer makes the system regulatory-ready.

The key principle:

No data access without identity, consent, purpose, and audit.

### Consent engine

The consent engine should answer:

- Who is requesting access?
- What data do they want?
- Why do they need it?
- For how long?
- Can it be shared further?
- Can AI process it?
- Can it leave the local node?
- Can it be used for training?
- Can it be used anonymously for analytics?

### Policy engine

The policy engine controls rules.

Examples:

- Reception can see appointment details but not full medical history
- Dentist can see clinical record for assigned patient
- Marketing can see testimonials only after written consent
- AI agent can draft messages but cannot send without approval
- Payment agent can calculate invoice but cannot move funds without authorization
- External practitioner can only access shared files for a limited time

### Audit engine

Every important action should be logged.

Audit events should include:

- User ID
- Agent ID if applicable
- Device ID
- Action performed
- Data accessed
- Time
- Location/IP where appropriate
- Legal basis or consent reference
- Tool used
- Model used if AI was involved
- Human approval status
- Payment reference if applicable
- Hash of the event

The audit log should be tamper-evident.

This means the system can prove whether something changed after the fact.

## 12. Layer 7 - Sovereign AI agent runtime

The AI agent layer is where LUCA becomes alive.

The agent runtime should support:

- Local LLMs
- Cloud LLMs by user choice
- Retrieval from approved knowledge bases
- Tool calling
- Workflow automation
- Human approval steps
- Agent identity
- Agent permissions
- Agent spending limits
- Agent logs
- Offline operation where possible

### Local LLMs

Local models can power private workflows without sending sensitive data to the cloud.

Possible uses:

- Summarizing patient notes locally
- Drafting treatment plan explanations
- Searching clinic knowledge base
- Creating internal checklists
- Helping staff follow SOPs
- Organizing documents
- Translating patient instructions
- Drafting follow-ups
- Preparing education material

QVAC-style medical/health models can be explored where appropriate, but every model must be tested, bounded, and supervised.

### Autonomous workflow agents

Hermes/OpenClaw-style agents or local computer-use agents may eventually help run business workflows.

Examples:

- Prepare daily clinic briefing
- Draft patient follow-ups
- Organize new files
- Update CRM records
- Generate treatment plan PDF drafts
- Review missing consent forms
- Prepare invoice summaries
- Trigger reminders
- Prepare social media drafts
- Monitor operational tasks

Important rule:

Agents can assist, draft, organize, and recommend. High-risk actions require human approval.

### Agent safety levels

```text
Level 0 - Read-only assistant
Level 1 - Drafts and summaries
Level 2 - Internal workflow updates
Level 3 - External messages with human approval
Level 4 - Payment or record changes with explicit authorization
Level 5 - Fully autonomous actions only in low-risk, pre-approved workflows
```

Aura should begin at Levels 0-3.

## 18. Zero Trust security model

Zero Trust means never assuming trust just because a user, device, agent, or service is inside the network.

Every request must be verified.

### Zero Trust rules

- Verify user identity
- Verify device identity
- Verify agent identity
- Verify session
- Verify permission
- Verify purpose
- Verify consent
- Verify data scope
- Log the action
- Require approval when risk is high

### Practical Aura example

A staff member wants to access a patient record.

The system checks:

```text
Is this staff member active?
Is this device trusted?
Is this patient assigned to this clinic?
Does the staff role allow this access?
What data is needed?
Is there consent?
Is this action logged?
Is AI involved?
Does the AI agent have permission?
```

This is how privacy and compliance become operational, not theoretical.
